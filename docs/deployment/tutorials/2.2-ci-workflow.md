# Tutorial 2.2 & 2.3: CI ワークフローの作成 (バックエンド + フロントエンド)

このチュートリアルでは、GitHub Actions を使って、Pull Request (PR) を作成するたびにコードの品質チェックを自動で実行する CI (Continuous Integration) パイプラインを構築します。

## 前提知識

- GitHub リポジトリにコードが push できること
- Pull Request の作り方がわかること
- Tutorial 2.1 が完了していること（GitHub Secrets 設定済み）

## CI とは？

**Continuous Integration (継続的インテグレーション)** とは、コードの変更を頻繁にメインブランチに統合し、その都度自動テストを実行する開発手法です。

```
開発者がコード変更 → PR 作成 → CI が自動で実行
                                    ├── ビルド（コンパイルできるか？）
                                    ├── テスト（動作は正しいか？）
                                    └── Lint（コードスタイルは揃っているか？）
                              → 全て合格なら ✅ マージ可能
                              → 1つでも失敗なら ❌ 修正が必要
```

> **従来との違い**: 従来はデプロイ直前に手動でビルドし、テストは「時間がないから省略」されることが多かった。CIでは全てのコード変更が自動的にビルド・テストされ、問題がある変更はマージ自体ができない。

## この手順でできること

PR を作成すると、以下が自動で実行されます:

| ジョブ | 内容 | 所要時間 |
|--------|------|---------|
| Backend Build & Test | Java (Maven) のコンパイル + 単体テスト | ~50 秒 |
| Frontend Build & Test | Angular の依存解決 + Lint + テスト + ビルド | ~90 秒 |

## 所要時間

約 5〜10 分（ワークフローファイル作成 + 動作確認）

## Step 1: ワークフローファイルの作成

GitHub Actions のワークフローは `.github/workflows/` ディレクトリに YAML ファイルとして配置します。

> **従来との違い**: 従来はJenkinsサーバーを立てて、GUIでジョブを設定していた。GitHub ActionsではYAMLファイルをリポジトリに含めるだけでCIが動く。Jenkinsの運用・メンテナンスが不要。

### 1.1 ディレクトリ作成

```bash
mkdir -p .github/workflows
```

### 1.2 CI ワークフローファイルの作成

`.github/workflows/ci.yml` を以下の内容で作成します:

```yaml
name: CI

on:
  pull_request:
    branches: [main]

jobs:
  backend:
    name: Backend Build & Test
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: proquip/proquip-parent
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: '17'
          cache: maven
          cache-dependency-path: proquip/proquip-parent/pom.xml

      - name: Build (compile + package)
        run: mvn clean package -B -DskipTests

      - name: Run unit tests (allow known failures)
        run: mvn test -B -Dmaven.test.failure.ignore=true

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: backend-test-results
          path: proquip/**/target/surefire-reports/*.xml
          retention-days: 7

  frontend:
    name: Frontend Build & Test
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: proquip/proquip-frontend
    steps:
      - uses: actions/checkout@v4

      - name: Set up Node.js 18
        uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: npm
          cache-dependency-path: proquip/proquip-frontend/package-lock.json

      - name: Install dependencies
        run: npm ci

      - name: Lint (non-blocking)
        run: npm run lint || true

      - name: Test (non-blocking)
        run: npm test -- --no-watch --browsers=ChromeHeadless || true

      - name: Build
        run: npm run build
```

### 各セクションの説明

#### トリガー設定

```yaml
on:
  pull_request:
    branches: [main]
```

`main` ブランチへの PR が作成・更新されたときに CI が起動します。

#### Backend ジョブ

| ステップ | 説明 |
|---------|------|
| `actions/checkout@v4` | リポジトリのコードを CI サーバーにダウンロード |
| `actions/setup-java@v4` | JDK 17 をインストール（Maven ビルドに必要） |
| `mvn clean package -B -DskipTests` | Java コードをコンパイルし、EAR パッケージを作成 |
| `mvn test -B -Dmaven.test.failure.ignore=true` | 単体テストを実行（既知の失敗は許容） |
| `actions/upload-artifact@v4` | テスト結果 XML を GitHub にアップロード |

**`-Dmaven.test.failure.ignore=true` について**: このプロジェクトには意図的な技術的負債（壊れたテスト）が含まれています。テスト失敗でビルド全体を止めないようにしています。本番プロジェクトでは通常この設定は使いません。

> **従来との違い**: 従来は開発者のPCで`mvn package`を実行し、成功したEARファイルを「これでビルドできたので」と共有していた。CIでは全員が同じクリーンな環境でビルドされるため、「私のPCでは動くのに」問題が起きない。

#### Frontend ジョブ

| ステップ | 説明 |
|---------|------|
| `actions/checkout@v4` | リポジトリのコードをダウンロード |
| `actions/setup-node@v4` | Node.js 18 をインストール |
| `npm ci` | `package-lock.json` に基づいて依存パッケージをインストール |
| `npm run lint \|\| true` | コードスタイルチェック（非ブロッキング） |
| `npm test -- --no-watch --browsers=ChromeHeadless` | テスト実行（非ブロッキング） |
| `npm run build` | Angular アプリをビルド（これが CI ゲート） |

**`|| true` について**: lint とテストの既存の問題でビルドが止まらないようにしています。`npm run build`（コンパイル）が CI の合否判定ポイントです。

> **従来との違い**: 従来はフロントエンドのビルドは手動で`npm run build`を実行し、distフォルダをサーバーにコピーしていた。CIではビルド+lint+テストが自動実行され、コード品質が一定以上保たれる。

```bash

git push origin your-branch
```

## Step 3: PR を作成して CI を確認

### 3.1 PR の作成

```bash
# GitHub CLI を使う場合
gh pr create --base main --title "your PR title" --body "description"

# または GitHub の Web UI から作成
```

### 3.2 CI の実行を確認

PR ページの下部に CI のステータスが表示されます:

```
✅ Backend Build & Test — passed
✅ Frontend Build & Test — passed
```

GitHub CLI で確認する場合:

```bash
# PR のチェック状態を確認
gh pr checks <PR番号>

# CI の実行ログをリアルタイムで見る
gh run watch <run-id>
```

### 3.3 CI が失敗した場合

失敗したジョブをクリックすると、どのステップで失敗したかログが見られます。

よくある失敗原因:

| 失敗箇所 | 原因 | 対処 |
|---------|------|------|
| Build (compile) | Java のコンパイルエラー | コードを修正して再 push |
| Build (npm run build) | Angular のコンパイルエラー | TypeScript エラーを修正 |
| Set up JDK / Node.js | バージョン指定ミス | ci.yml のバージョンを確認 |

> **従来との違い**: 従来はコードレビューは任意で、レビューなしで本番にデプロイされることもあった。CIではPRのステータスチェックが赤なら物理的にマージできないため、品質ゲートとして機能する。

## ワークフローのカスタマイズ

### テストを厳密にしたい場合

技術的負債を解消した後、テスト失敗でビルドを止めるようにします:

```yaml
# Backend: -Dmaven.test.failure.ignore=true を削除
- name: Run unit tests
  run: mvn test -B

# Frontend: || true を削除
- name: Lint
  run: npm run lint

- name: Test
  run: npm test -- --no-watch --browsers=ChromeHeadless
```

### 特定のディレクトリの変更時だけ CI を実行

```yaml
on:
  pull_request:
    branches: [main]
    paths:
      - 'proquip/**'
      - '.github/workflows/ci.yml'
```

### キャッシュを活用して高速化

上記の ci.yml では `cache: maven` と `cache: npm` を設定済み。2 回目以降の実行では依存パッケージのダウンロードがスキップされ、高速になります。

## 用語集

| 用語 | 説明 |
|------|------|
| CI | Continuous Integration — コード変更ごとに自動テストを実行する仕組み |
| GitHub Actions | GitHub が提供する CI/CD サービス。ワークフローを YAML で定義 |
| ワークフロー | CI/CD の一連の処理を定義したファイル（`.github/workflows/*.yml`） |
| ジョブ | ワークフロー内の実行単位。並列実行可能 |
| ステップ | ジョブ内の個々の処理 |
| Runner | CI を実行するサーバー。`ubuntu-latest` は GitHub が提供する Linux サーバー |
| Artifact | CI の実行結果（テスト報告書など）を保存したファイル |
| Maven | Java プロジェクトのビルドツール |
| npm | Node.js のパッケージマネージャー |
| Lint | コードスタイルの自動チェック |
| ChromeHeadless | 画面なしで動く Chrome ブラウザ。CI でのテスト実行に使う |
