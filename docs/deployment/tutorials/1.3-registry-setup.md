# Tutorial 1.3: コンテナレジストリの設定

このチュートリアルでは、OpenShift の内部コンテナレジストリを外部からアクセス可能にし、コンテナイメージを push/pull できるようにします。

## 前提知識

- Tutorial 1.2 が完了していること（`oc login` でクラスタに接続済み）
- コンテナイメージの基本概念がわかること（わからなくても手順どおりに進められます）

## コンテナレジストリとは？

コンテナレジストリは、コンテナイメージ（アプリケーションの実行パッケージ）を保管する場所です。

身近な例えでいうと:
- **GitHub** がソースコードの保管場所
- **コンテナレジストリ** がビルド済みアプリの保管場所

有名なレジストリ:
- Docker Hub（公開イメージが多い）
- Amazon ECR（AWS のプライベートレジストリ）
- **OpenShift 内部レジストリ**（今回使うもの — クラスタに内蔵されている）

## なぜ OpenShift 内部レジストリを使うのか？

| 選択肢 | メリット | デメリット |
|--------|---------|----------|
| **OpenShift 内部レジストリ** | 追加コストなし、クラスタ内の通信が高速、設定が簡単 | クラスタが停止するとレジストリも停止 |
| Amazon ECR | 高可用性、クラスタ外からもアクセス可能 | 追加コスト、ECR トークンの定期更新が必要 |

Dev/Staging 環境では内部レジストリで十分です。

## 所要時間

約 5〜10 分（クラスタの再起動待ちを含む）

## Step 1: 内部レジストリの状態確認

まず、内部レジストリが動いているか確認します。

```bash
oc get configs.imageregistry.operator.openshift.io cluster \
  -o jsonpath='{.spec.managementState}'
```

**期待する出力**: `Managed`

- `Managed` → レジストリは有効（次のステップへ）
- `Removed` → レジストリは無効化されている（有効化が必要）

レジストリが無効の場合:

```bash
oc patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"managementState":"Managed"}}'
```

レジストリの準備状況を確認:

```bash
oc get configs.imageregistry.operator.openshift.io cluster \
  -o jsonpath='{.status.conditions[?(@.type=="Available")].status}'
```

`True` と表示されれば準備完了です。

> **従来との違い**: 従来はEARやWARファイルを直接サーバーにSCP/FTPでコピーしていた。コンテナレジストリは「ビルド済みアプリのパッケージ倉庫」で、どのバージョンでもすぐにデプロイ・ロールバックできる。

---

## Step 2: レジストリを外部に公開する

デフォルトでは内部レジストリはクラスタ内部からしかアクセスできません。
CI/CD パイプライン（GitHub Actions など）からイメージを push するには、外部からアクセスできるようにする必要があります。

```bash
oc patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"defaultRoute":true}}'
```

**重要**: このコマンド実行後、クラスタの API サーバーが一時的に再起動する場合があります（特にシングルノード環境）。1〜2 分ほど `oc` コマンドが応答しなくなることがありますが、正常な動作です。待ってから再試行してください。

公開されたルート（URL）を確認:

```bash
oc get route -n openshift-image-registry
```

**成功例**:
```
NAME            HOST/PORT                                                                     SERVICES         PORT    TERMINATION
default-route   default-route-openshift-image-registry.apps.cluster-xxx.sandbox.opentlc.com   image-registry   <all>   reencrypt
```

`HOST/PORT` 列に表示された URL がレジストリのアドレスです。メモしておいてください。

> **従来との違い**: 従来はJenkins等のCIサーバーからSCPで直接サーバーに配置していた。レジストリ経由にすることで、ビルドとデプロイが分離され、同じイメージを複数環境にデプロイできる。

---

## Step 3: レジストリへのログイン

レジストリにイメージを push するには、まずログインが必要です。

```bash
# レジストリ URL を変数に保存
REGISTRY_URL=$(oc get route default-route -n openshift-image-registry \
  -o jsonpath='{.spec.host}')

echo "Registry URL: ${REGISTRY_URL}"

# ログイン（OpenShift のトークンを使用）
TOKEN=$(oc whoami -t)
podman login "${REGISTRY_URL}" \
  --username unused \
  --password "${TOKEN}" \
  --tls-verify=false
```

**成功例**:
```
Login Succeeded!
```

**パラメータの説明**:

| パラメータ | 値 | 説明 |
|-----------|-----|------|
| `--username` | `unused` | OpenShift トークン認証では任意の値でよい |
| `--password` | `$(oc whoami -t)` | 現在の OpenShift ログイントークンをパスワードとして使用 |
| `--tls-verify=false` | — | 自己署名証明書の検証をスキップ（Dev 環境用） |

> **従来との違い**: 従来はSCPのパスワードやSSH鍵で認証。レジストリではトークン認証を使い、有効期限付きのアクセスで安全性が高い。

---

## Step 4: 動作確認（テスト push）

実際にイメージを push して、正しく動作するか確認します。

### 4.1 テスト用のプロジェクト作成

```bash
# proquip-dev プロジェクトを作成（または切り替え）
oc new-project proquip-dev 2>/dev/null || oc project proquip-dev
```

### 4.2 テストイメージを pull → タグ付け → push

```bash
# 小さなテストイメージをダウンロード
podman pull docker.io/library/busybox:1.36.1

# OpenShift レジストリ用にタグ付け
# 形式: <レジストリURL>/<プロジェクト名>/<イメージ名>:<タグ>
podman tag docker.io/library/busybox:1.36.1 \
  "${REGISTRY_URL}/proquip-dev/test-push:v1"

# push
podman push "${REGISTRY_URL}/proquip-dev/test-push:v1" --tls-verify=false
```

### 4.3 push されたことを確認

```bash
oc get imagestream -n proquip-dev
```

**成功例**:
```
NAME        IMAGE REPOSITORY                                                       TAGS   UPDATED
test-push   default-route-openshift-image-registry.apps.cluster-xxx.../test-push   v1     7 seconds ago
```

`test-push` が表示されれば成功です。

### 4.4 テストイメージの削除

確認が終わったらテストイメージを削除します。

```bash
oc delete imagestream test-push -n proquip-dev
podman rmi docker.io/library/busybox:1.36.1
```

> **従来との違い**: 従来はデプロイ前のテストは手動で本番サーバーにファイルを置いて確認。レジストリではイメージをpushしてからデプロイは別ステップなので、安全に確認できる。

## Step 5: イメージの命名規則を理解する

OpenShift 内部レジストリにイメージを push する際の命名規則:

```
<レジストリURL>/<プロジェクト名>/<イメージ名>:<タグ>
```

ProQuip で使う 3 つのイメージ:

| イメージ | フルパス |
|---------|---------|
| WildFly | `<REGISTRY_URL>/proquip-dev/proquip-wildfly:latest` |
| nginx | `<REGISTRY_URL>/proquip-dev/proquip-nginx:latest` |
| Keycloak | `<REGISTRY_URL>/proquip-dev/proquip-keycloak:latest` |

OpenShift の Deployment からは、クラスタ内部の短いアドレスで参照できます:

```
image-registry.openshift-image-registry.svc:5000/proquip-dev/proquip-wildfly:latest
```

> **従来との違い**: 従来はproquip-1.0.ear, proquip-1.1.earのようなファイル名でバージョン管理。コンテナではタグ(latest, v1.0, commit-sha)で管理し、イメージの履歴が自動的に残る。

## 完了チェック

以下が全て成功すれば完了です:

```bash
# 1. レジストリが公開されているか
oc get route default-route -n openshift-image-registry
# → HOST/PORT が表示される

# 2. ログインできるか
REGISTRY_URL=$(oc get route default-route -n openshift-image-registry \
  -o jsonpath='{.spec.host}')
podman login "${REGISTRY_URL}" \
  --username unused \
  --password "$(oc whoami -t)" \
  --tls-verify=false
# → Login Succeeded!
```

## トラブルシューティング

### `oc` コマンドが応答しない

レジストリ設定変更後に API サーバーが再起動している可能性があります。
1〜2 分待ってから再試行してください。

### `Login Succeeded!` が出ない

```bash
# トークンが有効か確認
oc whoami
# → ユーザー名が表示されれば有効

# トークンを再取得してリトライ
TOKEN=$(oc whoami -t)
podman login "${REGISTRY_URL}" --username unused --password "${TOKEN}" --tls-verify=false
```

### push 時に `denied: requested access to the resource is denied`

push 先のプロジェクトに対する権限がない可能性があります:

```bash
# 現在のプロジェクトを確認
oc project

# プロジェクトに対する権限を確認
oc auth can-i create imagestreams -n proquip-dev
# → yes と表示されるべき
```

### `podman` コマンドが見つからない

Docker でも同じ操作が可能です。`podman` を `docker` に置き換えてください:

```bash
docker login "${REGISTRY_URL}" --username unused --password "${TOKEN}"
docker push "${REGISTRY_URL}/proquip-dev/test-push:v1"
```

## 用語集

| 用語 | 説明 |
|------|------|
| コンテナレジストリ | コンテナイメージを保管・配布するサーバー |
| コンテナイメージ | アプリケーションと実行環境をまとめたパッケージ |
| ImageStream | OpenShift がイメージを管理する仕組み。タグの変更を追跡できる |
| Route | OpenShift でサービスを外部に公開する仕組み（Kubernetes の Ingress に相当） |
| TLS | 通信の暗号化の仕組み。reencrypt = Route とレジストリの間も暗号化 |
| podman | Docker と互換性のあるコンテナツール。root 権限不要 |
