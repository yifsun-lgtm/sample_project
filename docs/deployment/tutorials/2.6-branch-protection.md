# Tutorial 2.6: Branch Protection Rule の設定

このチュートリアルでは、main ブランチを保護するルールを設定し、CI が通った PR のみがマージできるようにします。

## 前提知識

- GitHub リポジトリの Admin 権限があること
- Tutorial 2.2/2.3 (CI ワークフロー) が完了していること

## Branch Protection とは？

main ブランチに直接 push できないようにし、必ず PR 経由 + CI パス + レビュー承認を経てからマージするルールです。

```
保護なし:
  developer → git push main  ← 誰でも直接 main に push できる（危険）

保護あり:
  developer → feature branch → PR → CI パス ✅ → レビュー承認 ✅ → マージ
                                      ↑ 自動             ↑ 人間
```

> **従来との違い**: 従来はmainブランチ（本番コード）に誰でも直接pushできた。「うっかり本番を壊す」事故が起きていた。Branch protectionは「テストを通り、レビューされたコードだけが本番に入る」ことを技術的に強制する。

## 設定する保護ルール

| ルール | 設定値 | 効果 |
|--------|-------|------|
| Require status checks | Backend Build & Test, Frontend Build & Test | CI が通らないとマージ不可 |
| Require branches to be up to date | Yes | マージ前に main の最新を取り込む必要あり |
| Require pull request reviews | 1 approval | 最低 1 人のレビュー承認が必要 |
| Enforce admins | Yes | 管理者も同じルールに従う |
| Allow force pushes | No | 履歴の書き換えを禁止 |

## 所要時間

約 3 分

---

## 方法 A: GitHub CLI を使う方法（推奨）

```bash
gh api repos/your-username/your-repo/branches/main/protection \
  --method PUT \
  --input - <<'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["Backend Build & Test", "Frontend Build & Test"]
  },
  "required_pull_request_reviews": {
    "required_approving_review_count": 1
  },
  "enforce_admins": true,
  "restrictions": null
}
EOF
```

**パラメータの説明**:

| パラメータ | 値 | 意味 |
|-----------|-----|------|
| `strict: true` | — | PR ブランチが main の最新でないとマージ不可 |
| `contexts` | CI ジョブ名の配列 | これらのステータスチェックが全て緑でないとマージ不可 |
| `required_approving_review_count` | `1` | 最低 1 人の承認が必要 |
| `enforce_admins` | `true` | 管理者もルールを守る必要あり |
| `restrictions` | `null` | push 制限なし（PR 経由ルールで十分） |

### 設定確認

```bash
gh api repos/your-username/your-repo/branches/main/protection \
  | jq '{
    status_checks: .required_status_checks.contexts,
    strict_updates: .required_status_checks.strict,
    required_approvals: .required_pull_request_reviews.required_approving_review_count,
    enforce_admins: .enforce_admins.enabled,
    force_push: .allow_force_pushes.enabled
  }'
```

**期待する出力**:
```json
{
  "status_checks": ["Backend Build & Test", "Frontend Build & Test"],
  "strict_updates": true,
  "required_approvals": 1,
  "enforce_admins": true,
  "force_push": false
}
```

## 方法 B: GitHub Web UI を使う方法

1. GitHub でリポジトリページを開く
2. **Settings** → **Branches** をクリック
3. **Add branch protection rule** をクリック
4. 以下を設定:

| 設定項目 | 操作 |
|---------|------|
| Branch name pattern | `main` と入力 |
| Require a pull request before merging | ✅ チェック |
| Required approvals | `1` を選択 |
| Require status checks to pass before merging | ✅ チェック |
| Search for status checks | `Backend Build & Test` と `Frontend Build & Test` を検索して追加 |
| Require branches to be up to date before merging | ✅ チェック |
| Include administrators | ✅ チェック |

5. **Create** をクリック

**注意**: ステータスチェック名は CI が 1 回以上実行された後でないと検索に出てきません。

> **従来との違い**: 従来はデプロイ前のレビューは「口頭確認」や「メールでの承認」で、形骸化しやすかった。Branch protectionではCIパス+レビュー承認がないと物理的にマージできないため、確実にゲートが機能する。

## 動作確認

### Direct push が拒否されるか確認

```bash
# main ブランチで直接 push してみる（失敗するはず）
git checkout main
echo "test" >> test.txt
git add test.txt
git commit -m "test direct push"
git push origin main
# → error: protected branch hook declined
```

### CI 未パスの PR がマージ不可か確認

CI が失敗する PR を作ると、マージボタンがグレーアウトします。

> **従来との違い**: 従来は「本番に直接pushしないでください」というルールがドキュメントに書かれていたが、技術的に防止する仕組みがなかった。Branch protectionは「人の注意力」ではなく「システム」でミスを防ぐ。

---

## 完了チェック

以下の 3 点が確認できれば完了:

1. main への direct push が拒否される
2. CI が赤の PR はマージ不可
3. approve がない PR はマージ不可

```bash
# CLI で確認
gh api repos/your-username/your-repo/branches/main/protection \
  | jq '.required_status_checks.contexts, .required_pull_request_reviews.required_approving_review_count'
```

---

## 保護ルールの一時解除（緊急時のみ）

本番障害などで緊急マージが必要な場合:

```bash
# 保護を削除
gh api repos/your-username/your-repo/branches/main/protection --method DELETE

# 緊急マージ実行後、保護を再設定
# (方法 A のコマンドを再実行)
```

**注意**: `enforce_admins: true` にしているため、管理者でもルールをバイパスできません。緊急時は保護自体を一時的に削除する必要があります。

## トラブルシューティング

### ステータスチェックが検索に出てこない

CI ワークフローが 1 回も実行されていない可能性があります。まず PR を作成して CI を実行してください。

### `Resource not accessible by integration` エラー

リポジトリの Admin 権限がありません。リポジトリのオーナーに依頼してください。

### 保護ルールを変更したい

```bash
# 現在の設定を確認
gh api repos/your-username/your-repo/branches/main/protection | jq .

# 設定を上書き（同じ PUT コマンドを新しい値で実行）
```

---

## 用語集

| 用語 | 説明 |
|------|------|
| Branch protection | ブランチへの変更を制限するルール |
| Status check | CI の実行結果。passed/failed/pending のいずれか |
| Required reviewers | PR をマージする前に承認が必要な人数 |
| Enforce admins | 管理者にも同じルールを適用するかどうか |
| Force push | コミット履歴を書き換えて push する操作。通常は禁止 |
| strict | マージ前に PR ブランチを main の最新に更新する必要があるかどうか |
