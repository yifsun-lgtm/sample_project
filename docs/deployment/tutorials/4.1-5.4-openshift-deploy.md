# Tutorial 4.1–5.4: OpenShift マニフェスト作成とデプロイ検証

このチュートリアルでは、ProQuip の各コンポーネントを OpenShift にデプロイし、動作を検証します。

## 前提知識

- Phase 1–3 が完了していること
- `oc login` でクラスタに接続済み
- コンテナイメージが OpenShift レジストリに push 済み

## この手順でできること

| コンポーネント | 役割 |
|--------------|------|
| WildFly | Jakarta EE バックエンド API サーバー |
| Keycloak | 認証・認可サーバー (OIDC) |
| nginx | フロントエンド (Angular SPA) 配信 + リバースプロキシ |
| Flyway | データベースマイグレーション (Job) |

## 所要時間

約 30〜60 分 (イメージビルド待ち含む)

---

## マニフェストファイル構成

```
k8s/
├── 00-namespace.yaml          # プロジェクト
├── 01-pull-secret.yaml         # レジストリ認証 (内部レジストリの場合は不要)
├── 02-configmap.yaml           # 環境変数 (DB接続先, Keycloak設定)
├── 03-secrets.yaml             # 機密情報 (DBパスワード等) ← oc CLI で作成
├── 10-flyway-job.yaml          # DBマイグレーション
├── 20-wildfly-deployment.yaml  # WildFly + Service
├── 30-keycloak-deployment.yaml # Keycloak + Service
├── 38-keycloak-realm-import.yaml # レルムインポート Job
└── 40-nginx-deployment.yaml    # nginx + Service + Route
```

> **従来との違い**: 従来はサーバーの設定がドキュメント(手順書)にしか残っておらず、設定が古くなりがちだった。k8s/マニフェストはインフラの状態をコードとして管理(Infrastructure as Code)するため、Gitで変更履歴を追跡でき、誰がいつ何を変えたかが明確。

## Step 1: Namespace (プロジェクト) の作成

```bash
oc new-project proquip-dev
# または
oc apply -f k8s/00-namespace.yaml
```

> **従来との違い**: 従来は1台のサーバーに複数のアプリを同居させることが多く、設定ファイルやログが混在していた。Namespaceはアプリごとの論理的な隔離を提供し、リソースの競合を防ぐ。

---

## Step 2: ConfigMap の作成

`k8s/02-configmap.yaml` にデータベース接続先や Keycloak の設定を記述します。

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: proquip-config
  namespace: proquip-dev
data:
  DB_HOST: "your-rds-endpoint.rds.amazonaws.com"
  DB_PORT: "5432"
  DB_NAME: "proquip"
  KEYCLOAK_URL: "http://proquip-keycloak:8080"
  KEYCLOAK_REALM: "proquip"
  KEYCLOAK_CLIENT_ID: "proquip-api"
  KC_DB: "postgres"
  KC_DB_URL: "jdbc:postgresql://your-rds-endpoint.rds.amazonaws.com:5432/keycloak"
  KC_HOSTNAME_STRICT: "false"
  KC_PROXY: "edge"
  KC_HTTP_ENABLED: "true"
```

```bash
oc apply -f k8s/02-configmap.yaml
```

> **従来との違い**: 従来はstandalone.xmlやapplication.propertiesをサーバー上で直接編集していた。ConfigMapは設定をアプリケーションコードから分離し、同じイメージを異なる設定で複数環境にデプロイできる。

## Step 3: Secret の作成

**重要**: Secret は YAML ファイルに書かずに、`oc` コマンドで直接作成します。

```bash
oc create secret generic proquip-secrets -n proquip-dev \
  --from-literal=DB_USER=proquip_admin \
  --from-literal=DB_PASSWORD='your-db-password' \
  --from-literal=KC_DB_USERNAME=proquip_admin \
  --from-literal=KC_DB_PASSWORD='your-db-password' \
  --from-literal=KEYCLOAK_ADMIN=admin \
  --from-literal=KEYCLOAK_ADMIN_PASSWORD='your-keycloak-admin-password'
```

> **従来との違い**: 従来はDBパスワードが設定ファイルに平文で書かれ、Gitにコミットされていることもあった。SecretはBase64エンコード+RBAC制御で保護され、`oc get secret`でも値は表示されない。

## Step 4: Flyway マイグレーション

Flyway 用のカスタムイメージを作成して、データベースマイグレーションを実行します。

### Flyway イメージのビルド

```bash
# amd64 でビルド (OpenShift ノードが x86_64 の場合)
podman build --platform linux/amd64 \
  -f docker/flyway/Dockerfile \
  -t <REGISTRY>/proquip-dev/proquip-flyway:latest .
podman push <REGISTRY>/proquip-dev/proquip-flyway:latest --tls-verify=false
```

**注意**: macOS (Apple Silicon) でビルドする場合、`--platform linux/amd64` を必ず付けてください。付けないと OpenShift 上で `exec format error` になります。

### マイグレーション実行

```bash
oc apply -f k8s/10-flyway-job.yaml
oc wait --for=condition=complete job/proquip-flyway-migrate -n proquip-dev --timeout=120s
oc logs job/proquip-flyway-migrate -n proquip-dev
```

成功例:
```
Successfully applied 22 migrations to schema "public", now at version v030
```

> **従来との違い**: 従来はDBAが手動でSQLを実行し、「どのSQLまで実行したか」をExcelで管理していた。Flyway Jobは冪等性があり、何度実行しても同じ結果になる。実行済みのマイグレーションはスキップされる。

## Step 5: アプリケーションのデプロイ

### Keycloak

```bash
oc apply -f k8s/30-keycloak-deployment.yaml
```

### WildFly

```bash
oc apply -f k8s/20-wildfly-deployment.yaml
```

### nginx + Route

```bash
oc apply -f k8s/40-nginx-deployment.yaml
```

### Pod の状態確認

```bash
oc get pods -n proquip-dev -l 'app in (proquip-wildfly,proquip-keycloak,proquip-nginx)'
```

全ての Pod が `Running` かつ `READY 1/1` になるまで待ちます:

```
NAME                                READY   STATUS    AGE
proquip-keycloak-xxxxx-xxxxx        1/1     Running   2m
proquip-nginx-xxxxx-xxxxx           1/1     Running   2m
proquip-wildfly-xxxxx-xxxxx         1/1     Running   2m
```

> **従来との違い**: 従来はWildFlyを停止→EAR差替え→起動で、その間サービス停止。OpenShiftのローリングアップデートでは旧Podが動いたまま新Podが起動し、準備完了後に切り替わるためダウンタイムゼロ。

## Step 6: アクセス確認

### Route URL の取得

```bash
oc get route proquip -n proquip-dev -o jsonpath='{.spec.host}'
```

### エンドポイントの確認

```bash
ROUTE="https://$(oc get route proquip -n proquip-dev -o jsonpath='{.spec.host}')"

# フロントエンド
curl -sk -o /dev/null -w 'Frontend: HTTP %{http_code}\n' "${ROUTE}/"

# ヘルスチェック
curl -sk "${ROUTE}/health.json"

# API ヘルスチェック
curl -sk "${ROUTE}/api/health"

# Keycloak OIDC
curl -sk -o /dev/null -w 'Keycloak: HTTP %{http_code}\n' \
  "${ROUTE}/realms/proquip/.well-known/openid-configuration"
```

### ユーザーログインテスト

```bash
KC_TOKEN_URL="${ROUTE}/realms/proquip/protocol/openid-connect/token"

curl -sk -X POST "${KC_TOKEN_URL}" \
  -d "client_id=proquip-web" \
  -d "grant_type=password" \
  -d "username=admin" \
  -d "password=admin123"
```

`access_token` が返ってくれば成功です。

> **従来との違い**: 従来はデプロイ後に手動でブラウザを開いて目視確認。CI/CDではヘルスチェックが自動実行され、異常があれば自動ロールバックも可能。

テストユーザー一覧:

| ユーザー | パスワード | ロール |
|---------|----------|--------|
| admin | admin123 | ADMIN |
| manager | manager123 | MANAGER |
| buyer | buyer123 | BUYER |
| warehouse | warehouse123 | WAREHOUSE_STAFF |
| viewer | viewer123 | VIEWER |

## トラブルシューティング

### Pod が Pending のまま

```bash
oc describe pod <pod-name> -n proquip-dev | grep -A5 Events
```

`Insufficient cpu/memory` の場合:
- Deployment から `resources.requests` と `resources.limits` を削除
- シングルノードクラスタではリソース制約が厳しい

### `exec format error`

イメージのアーキテクチャが OpenShift ノードと異なります。

```bash
# ノードのアーキテクチャ確認
oc get node -o jsonpath='{.items[0].status.nodeInfo.architecture}'
# → amd64

# amd64 でリビルド
podman build --platform linux/amd64 ...
```

### nginx が `host not found in upstream`

nginx.conf の upstream 名が OpenShift の Service 名と一致していることを確認:

```nginx
upstream wildfly_backend {
    server proquip-wildfly:8080;  # ← Service 名
}
upstream keycloak_backend {
    server proquip-keycloak:8080;  # ← Service 名
}
```

### WildFly が `Directory ... is not writable`

Dockerfile で OpenShift 対応 (chgrp/chmod) が漏れています:

```dockerfile
USER root
RUN chgrp -R 0 /opt/jboss/wildfly && chmod -R g=u /opt/jboss/wildfly
USER 1001
```

## 用語集

| 用語 | 説明 |
|------|------|
| Deployment | Pod のレプリカを管理するリソース。更新時にローリングアップデート |
| Service | Pod へのネットワークアクセスを抽象化。DNS 名で他の Pod から接続可能 |
| Route | Service を外部に公開する OpenShift 固有のリソース (Ingress に相当) |
| ConfigMap | 機密でない設定情報を Key-Value で管理するリソース |
| Secret | パスワード等の機密情報を暗号化して管理するリソース |
| Job | 一度だけ実行するタスク (マイグレーション等) |
| readinessProbe | Pod が準備完了かを判定するヘルスチェック |
| livenessProbe | Pod が正常動作中かを判定するヘルスチェック |
