# Tutorial 2.4 & 2.5: CD ワークフローの作成 (イメージビルド + OpenShift デプロイ)

このチュートリアルでは、GitHub Actions を使って、コードが main ブランチにマージされるたびにコンテナイメージのビルド・プッシュと OpenShift への自動デプロイを実行する CD (Continuous Delivery) パイプラインを構築します。

## 前提知識

- Tutorial 2.2 (CI ワークフロー) が完了していること
- Tutorial 2.1 (GitHub Secrets) が設定済みであること
- Tutorial 1.3 (OpenShift レジストリ) が設定済みであること

## CD とは？

**Continuous Delivery (継続的デリバリー)** とは、コードの変更がテストを通過したら、自動的に本番環境（またはステージング環境）にデプロイする仕組みです。

```
PR マージ → main push → CD が自動実行
                          ├── イメージビルド (WildFly, nginx, Keycloak)
                          ├── レジストリに push
                          └── OpenShift にデプロイ → ヘルスチェック
```

## CI と CD の違い

| | CI | CD |
|--|-----|-----|
| トリガー | PR 作成・更新 | main へのマージ (push) |
| 目的 | コードの品質チェック | アプリケーションのデプロイ |
| 成果物 | テスト結果 | 動作するアプリケーション |
| ファイル | `ci.yml` | `cd.yml` |

> **従来との違い**: 従来はCIもCDも存在せず、全て手動だった。CIが「品質チェックの自動化」なら、CDは「デプロイ作業の自動化」。従来4-8時間かかっていたデプロイが5分で完了する。

## 所要時間

約 10〜15 分（ワークフロー作成 + 初回実行確認）

## Step 1: CD ワークフローファイルの作成

`.github/workflows/cd.yml` を以下の内容で作成します:

```yaml
name: CD

on:
  push:
    branches: [main]
    paths:
      - 'proquip/**'
      - '.github/workflows/cd.yml'

env:
  REGISTRY: ${{ secrets.REGISTRY_URL }}
  PROJECT: proquip-dev

jobs:
  build-and-push:
    name: Build & Push Images
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install oc CLI
        uses: redhat-actions/openshift-tools-installer@v1
        with:
          oc: '4'

      - name: Login to OpenShift
        run: |
          echo "${{ secrets.OPENSHIFT_CA_CERT }}" | base64 -d > /tmp/ca.crt
          oc login \
            --server=${{ secrets.OPENSHIFT_SERVER }} \
            --token=${{ secrets.OPENSHIFT_TOKEN }} \
            --certificate-authority=/tmp/ca.crt

      - name: Login to OpenShift Registry
        run: |
          oc whoami -t | docker login \
            ${{ env.REGISTRY }} \
            --username unused \
            --password-stdin

      - name: Set image tag
        id: tag
        run: |
          SHORT_SHA=$(echo "${{ github.sha }}" | cut -c1-7)
          echo "tag=${SHORT_SHA}" >> "$GITHUB_OUTPUT"

      - name: Build & Push WildFly
        working-directory: proquip
        run: |
          docker build -f docker/wildfly/Dockerfile \
            -t ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-wildfly:${{ steps.tag.outputs.tag }} \
            -t ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-wildfly:latest .
          docker push ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-wildfly:${{ steps.tag.outputs.tag }}
          docker push ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-wildfly:latest

      - name: Build & Push nginx
        working-directory: proquip
        run: |
          docker build -f docker/nginx/Dockerfile \
            -t ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-nginx:${{ steps.tag.outputs.tag }} \
            -t ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-nginx:latest .
          docker push ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-nginx:${{ steps.tag.outputs.tag }}
          docker push ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-nginx:latest

      - name: Build & Push Keycloak
        working-directory: proquip/docker/keycloak
        run: |
          docker build \
            -t ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-keycloak:${{ steps.tag.outputs.tag }} \
            -t ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-keycloak:latest .
          docker push ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-keycloak:${{ steps.tag.outputs.tag }}
          docker push ${{ env.REGISTRY }}/${{ env.PROJECT }}/proquip-keycloak:latest

    outputs:
      image-tag: ${{ steps.tag.outputs.tag }}

  deploy:
    name: Deploy to OpenShift
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install oc CLI
        uses: redhat-actions/openshift-tools-installer@v1
        with:
          oc: '4'

      - name: Login to OpenShift
        run: |
          echo "${{ secrets.OPENSHIFT_CA_CERT }}" | base64 -d > /tmp/ca.crt
          oc login \
            --server=${{ secrets.OPENSHIFT_SERVER }} \
            --token=${{ secrets.OPENSHIFT_TOKEN }} \
            --certificate-authority=/tmp/ca.crt

      - name: Update images
        run: |
          TAG="${{ needs.build-and-push.outputs.image-tag }}"
          NS="proquip-dev"
          INTERNAL_REG="image-registry.openshift-image-registry.svc:5000"

          for app in wildfly nginx keycloak; do
            if oc get deployment proquip-${app} -n ${NS} 2>/dev/null; then
              oc set image deployment/proquip-${app} \
                ${app}="${INTERNAL_REG}/${NS}/proquip-${app}:${TAG}" -n ${NS}
            else
              echo "Deployment proquip-${app} not found — skipping"
            fi
          done

      - name: Wait for rollout
        run: |
          NS="proquip-dev"
          for app in wildfly nginx keycloak; do
            if oc get deployment proquip-${app} -n ${NS} 2>/dev/null; then
              oc rollout status deployment/proquip-${app} -n ${NS} --timeout=180s
            fi
          done
```

### 各セクションの説明

#### トリガー設定

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'proquip/**'
      - '.github/workflows/cd.yml'
```

- `main` ブランチへの push（= PR のマージ）でのみ起動
- `paths` フィルター: `proquip/` 配下のファイルか `cd.yml` 自体が変更されたときだけ起動（ドキュメントだけの変更では起動しない）

> **従来との違い**: 従来はデプロイのタイミングは「担当者がメールで連絡して、深夜に実施」だった。CDではmainブランチへのマージが自動的にデプロイをトリガーする。深夜作業も立会いも不要。

#### イメージタグ戦略

```yaml
- name: Set image tag
  id: tag
  run: |
    SHORT_SHA=$(echo "${{ github.sha }}" | cut -c1-7)
    echo "tag=${SHORT_SHA}" >> "$GITHUB_OUTPUT"
```

各イメージには 2 つのタグを付けます:
- **commit SHA** (例: `83027e3`): 特定のコミットに紐づく不変のタグ
- **`latest`**: 常に最新のイメージを指す

> **従来との違い**: 従来はproquip-v1.2.3.earのようなファイル名でバージョン管理し、どのソースコードからビルドされたか追跡困難だった。commit SHAタグならGitの履歴と完全に紐付き、「このバージョンはどのコミットか」が一目瞭然。

#### ビルドコンテキストの注意

```yaml
# WildFly と nginx は proquip/ をコンテキストにする
working-directory: proquip
run: docker build -f docker/wildfly/Dockerfile ...

# Keycloak は docker/keycloak/ をコンテキストにする
# (proquip-realm.json が docker/keycloak/ 内にあるため)
working-directory: proquip/docker/keycloak
run: docker build ...
```

Dockerfile 内の `COPY` 命令はビルドコンテキスト（`working-directory`）からの相対パスでファイルを探します。Keycloak の Dockerfile は `COPY proquip-realm.json ...` と書いてあるため、ビルドコンテキストを `proquip-realm.json` がある `docker/keycloak/` に合わせます。

> **従来との違い**: 従来はビルドスクリプトがサーバー固有のパスに依存していた（/opt/deploy/scripts/build.sh等）。Dockerfileとcd.ymlで全てのビルド手順がコード化され、誰でも再現可能。

#### デプロイジョブ

```yaml
deploy:
  needs: build-and-push  # build-and-push が成功してから実行
```

- `needs`: ジョブ間の依存関係。イメージが push されてからデプロイ
- `oc set image`: Deployment のコンテナイメージを更新
- `oc rollout status`: 新しい Pod が起動完了するまで待機

#### 内部レジストリ URL の使い分け

| 場面 | URL | 説明 |
|------|-----|------|
| CI/CD から push | `default-route-openshift-image-registry.apps.cluster-xxx...` | 外部ルート（インターネット経由） |
| Deployment で参照 | `image-registry.openshift-image-registry.svc:5000` | クラスタ内部 URL（高速） |

> **従来との違い**: 従来は「WildFlyを停止→EARを差し替え→WildFlyを起動」で、その間サービス停止していた。OpenShiftのローリングアップデートでは旧Podが動いたまま新Podが起動するため、ダウンタイムゼロ。

## Step 2: 動作確認

### 2.1 コミットしてプッシュ

```bash
git add .github/workflows/cd.yml
git commit -m "Add CD workflow for image build/push and OpenShift deploy"
git push origin your-branch
```

### 2.2 PR を作成してマージ

```bash
gh pr create --base main --title "Add CD workflow"
# CI が通ったら:
gh pr merge --squash
```

### 2.3 CD の実行を確認

```bash
# CD ワークフローの実行を確認
gh run list --workflow=cd.yml --limit=1

# リアルタイムでログを確認
gh run watch <run-id>
```

### 2.4 OpenShift でイメージを確認

```bash
oc get imagestream -n proquip-dev
```

**成功例**:
```
NAME               TAGS                  UPDATED
proquip-wildfly    latest,83027e3        2 minutes ago
proquip-nginx      latest,83027e3        1 minute ago
proquip-keycloak   latest,83027e3        30 seconds ago
```

## ロールバック

デプロイ後に問題が見つかった場合の手動ロールバック:

```bash
# 直前のバージョンに戻す
oc rollout undo deployment/proquip-wildfly -n proquip-dev
oc rollout undo deployment/proquip-nginx -n proquip-dev
oc rollout undo deployment/proquip-keycloak -n proquip-dev

# 特定のリビジョンに戻す
oc rollout history deployment/proquip-wildfly -n proquip-dev
oc rollout undo deployment/proquip-wildfly --to-revision=2 -n proquip-dev
```

> **従来との違い**: 従来のロールバックは「バックアップしておいたEARファイルを探して差し替え、DBも戻して…」で1-2時間。`oc rollout undo`は30秒で完了し、失敗のストレスが激減。

## トラブルシューティング

### `COPY ... not found` エラー (Keycloak)

Dockerfile 内の `COPY` コマンドのソースファイルがビルドコンテキスト内に存在しません。
`working-directory` がファイルのある場所を指しているか確認してください。

### `unauthorized: authentication required` エラー

OpenShift トークンが期限切れの可能性があります:
1. 新しいトークンを取得: `oc whoami -t`
2. GitHub Secret を更新: `gh secret set OPENSHIFT_TOKEN --body "$(oc whoami -t)"`

### Deploy ジョブで `Deployment not found`

Phase 4 (OpenShift マニフェスト) がまだ完了していない場合の正常な動作です。
Deployment を作成すると、次回の CD 実行からデプロイが有効になります。

## 用語集

| 用語 | 説明 |
|------|------|
| CD | Continuous Delivery — コード変更を自動デプロイする仕組み |
| コンテナイメージ | アプリの実行パッケージ。Dockerfile からビルドされる |
| ImageStream | OpenShift がイメージを追跡する仕組み。タグの変更を検知できる |
| ビルドコンテキスト | `docker build` コマンドがファイルを探すルートディレクトリ |
| rollout | Deployment の Pod を新しいイメージで段階的に置き換えるプロセス |
| ロールバック | 問題があったときに前のバージョンに戻す操作 |
