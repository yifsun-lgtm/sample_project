# Tutorial 3.1–3.5: コンテナイメージの OpenShift 対応

このチュートリアルでは、ProQuip の 3 つの Docker イメージ (WildFly, Keycloak, nginx) を OpenShift で動作するように修正します。

## OpenShift が通常の Docker と異なる点

OpenShift はセキュリティのため、コンテナをランダムな非 root ユーザー ID で実行します。

| 項目 | Docker (ローカル) | OpenShift |
|------|-------------------|-----------|
| 実行ユーザー | root (UID 0) | ランダム UID (例: 1000620000) |
| グループ | root (GID 0) | root (GID 0) |
| ポート | 任意 (80 も可) | 1024 以上のみ |

つまり:
- ファイルはグループ (root) で書き込み可能にする必要がある
- 1024 未満のポート (80, 443 など) は使えない → 8080 に変更

> **従来との違い**: 従来のオンプレミスではアプリケーションはrootまたは専用ユーザーで実行し、ポート80/443を直接使用していた。OpenShiftのセキュリティ制約は面倒に見えるが、コンテナが乗っ取られても被害を最小限に抑えるための設計。

## 3.1: WildFly Dockerfile の修正

### 変更内容

Dockerfile の末尾 (EAR デプロイ後) に以下を追加:

```dockerfile
# OpenShift 対応: グループ書き込み権限 + 非 root ユーザー
USER root
RUN chgrp -R 0 /opt/jboss/wildfly && \
    chmod -R g=u /opt/jboss/wildfly
USER 1001
```

### 何をしているか

| コマンド | 意味 |
|---------|------|
| `chgrp -R 0` | 全ファイルのグループを root (GID 0) に変更 |
| `chmod -R g=u` | グループの権限をユーザーと同じにする (読み書き実行) |
| `USER 1001` | UID 1001 で実行 (OpenShift がランダム UID を割り当てるが、GID 0 のグループ権限でアクセス可能) |

### 確認方法

```bash
cd proquip
podman build -f docker/wildfly/Dockerfile -t proquip-wildfly:test .
podman run --rm --user 1001:0 proquip-wildfly:test id
# → uid=1001(1001) gid=0(root) groups=0(root)
```

> **従来との違い**: 従来はWildFlyをjbossユーザーで起動し、/opt/jboss/に全権限があった。OpenShift対応では「誰が実行しても動く」ようにファイル権限を設定する。これにより、特定のユーザーアカウントに依存しない可搬性の高いコンテナになる。

## 3.2: WildFly 設定の外部化

### 確認結果

`docker/wildfly/configure-wildfly.cli` は既に環境変数で外部化されていました:

| 環境変数 | デフォルト値 | 用途 |
|---------|------------|------|
| `DB_HOST` | `localhost` | データベースのホスト名 |
| `DB_PORT` | `5432` | データベースのポート |
| `DB_NAME` | `proquip` | データベース名 |
| `DB_USER` | `proquip` | データベースユーザー |
| `DB_PASSWORD` | `proquip` | データベースパスワード |
| `KEYCLOAK_URL` | `http://localhost:8180` | Keycloak の URL |
| `KEYCLOAK_REALM` | `proquip` | Keycloak のレルム名 |
| `KEYCLOAK_CLIENT_ID` | `proquip-api` | Keycloak のクライアント ID |

OpenShift の ConfigMap/Secret でこれらの環境変数を設定すれば RDS に接続できます。

> **従来との違い**: 従来はstandalone.xmlを手動編集し、サーバーごとに異なる設定ファイルを管理していた（本番用、検証用...）。環境変数での外部化により、同じイメージを全環境で使用でき、設定はConfigMap/Secretで管理される。

## 3.3: Keycloak Dockerfile の本番モード対応

### 変更前 (開発モード)

```dockerfile
FROM quay.io/keycloak/keycloak:22.0.5
COPY proquip-realm.json /opt/keycloak/data/import/proquip-realm.json
ENTRYPOINT ["/opt/keycloak/bin/kc.sh"]
CMD ["start-dev", "--import-realm"]
```

### 変更後 (本番モード)

```dockerfile
# Stage 1: ビルド (最適化)
FROM quay.io/keycloak/keycloak:22.0.5 AS builder
ENV KC_DB=postgres
ENV KC_HEALTH_ENABLED=true
RUN /opt/keycloak/bin/kc.sh build

# Stage 2: ランタイム
FROM quay.io/keycloak/keycloak:22.0.5
COPY --from=builder /opt/keycloak/ /opt/keycloak/
COPY proquip-realm.json /opt/keycloak/data/import/proquip-realm.json
ENTRYPOINT ["/opt/keycloak/bin/kc.sh"]
CMD ["start", "--optimized", "--hostname-strict=false", "--http-enabled=true", "--proxy=edge"]
```

### `start-dev` と `start --optimized` の違い

| | start-dev | start --optimized |
|--|-----------|-------------------|
| 用途 | 開発・テスト | 本番運用 |
| 起動速度 | 遅い (毎回ビルド) | 速い (事前ビルド済み) |
| HTTPS | 不要 | 必要 (proxy=edge で Route に委任) |
| DB ドライバー | 自動検出 | 事前にビルドで固定 |

### 必要な環境変数

| 環境変数 | 用途 |
|---------|------|
| `KC_DB_URL` | `jdbc:postgresql://<RDS_ENDPOINT>:5432/keycloak` |
| `KC_DB_USERNAME` | DB ユーザー名 |
| `KC_DB_PASSWORD` | DB パスワード |
| `KEYCLOAK_ADMIN` | 管理者ユーザー名 |
| `KEYCLOAK_ADMIN_PASSWORD` | 管理者パスワード |

> **従来との違い**: 従来はKeycloakをstart-devモードで本番運用しているケースも多かった。`start --optimized`は事前ビルドにより起動が高速化され、本番に適したセキュリティ設定が適用される。

## 3.4: nginx Dockerfile の OpenShift 対応

### nginx.conf の変更

```diff
- listen 80;
+ listen 8080;
```

### Dockerfile の変更

```dockerfile
# OpenShift 対応: 非 root 実行
RUN chgrp -R 0 /var/cache/nginx /var/run /var/log/nginx /usr/share/nginx/html && \
    chmod -R g=u /var/cache/nginx /var/run /var/log/nginx /usr/share/nginx/html && \
    sed -i '/^user  nginx;/d' /etc/nginx/nginx.conf && \
    sed -i 's|/var/run/nginx.pid|/tmp/nginx.pid|' /etc/nginx/nginx.conf

EXPOSE 8080

USER 1001
```

### 何をしているか

| 操作 | 理由 |
|------|------|
| `chgrp -R 0` + `chmod -R g=u` | キャッシュ・ログ・PID ファイルにグループ書き込み権限を付与 |
| `sed -i '/^user  nginx;/d'` | `user nginx;` ディレクティブを削除 (非 root では不要) |
| `sed -i 's\|/var/run/nginx.pid\|/tmp/nginx.pid\|'` | PID ファイルを書き込み可能な /tmp に移動 |
| `EXPOSE 8080` | ポート 80 → 8080 |
| `USER 1001` | 非 root で実行 |

### 確認方法

```bash
cd proquip
podman build -f docker/nginx/Dockerfile -t proquip-nginx:test .
podman run --rm --user 1001:0 \
  --add-host wildfly:127.0.0.1 --add-host keycloak:127.0.0.1 \
  -d --name nginx-test -p 8888:8080 proquip-nginx:test
curl http://localhost:8888/health.json
# → {"status":"ok"}
podman stop nginx-test
```

`--add-host` はローカルテスト用。OpenShift では Service 名で自動解決されます。

> **従来との違い**: 従来はnginxをrootで起動し、ポート80/443をlistenしていた。OpenShift対応でポート8080に変更するが、Routeが外部からのHTTPS(443)を受けて内部の8080に転送するため、ユーザーには違いが見えない。

## 3.5: ビルド・プッシュスクリプト

`scripts/build-push.sh` で 3 イメージをまとめてビルド・プッシュできます。

### 使い方

```bash
# デフォルト (latest タグ)
bash scripts/build-push.sh

# カスタムタグ
bash scripts/build-push.sh v1.0.0

# プロジェクト指定
bash scripts/build-push.sh v1.0.0 proquip-staging
```

### スクリプトがやること

1. OpenShift レジストリの URL を自動取得
2. `oc` トークンでレジストリにログイン
3. WildFly / nginx / Keycloak の順にビルド → プッシュ
4. 最後に ImageStream 一覧を表示して確認

> **従来との違い**: 従来はビルド手順が担当者のメモ帳やWikiに書かれており、手順が古くなりがちだった。スクリプト化によりビルド手順がバージョン管理され、誰でも同じ手順を再現できる。

## 用語集

| 用語 | 説明 |
|------|------|
| arbitrary UID | OpenShift がランダムに割り当てるユーザー ID。セキュリティのため root 以外で実行 |
| GID 0 | root グループ。OpenShift ではコンテナが必ず GID 0 に属する |
| `g=u` | グループの権限をユーザーと同じにする chmod の表記 |
| `--optimized` | Keycloak のビルド済み起動モード。起動が速い |
| multi-stage build | Dockerfile で複数の FROM を使い、ビルド環境とランタイムを分離する手法 |
| ImageStream | OpenShift がコンテナイメージのタグ変更を追跡する仕組み |
